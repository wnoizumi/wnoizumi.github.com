Revealing Design Problems in Stinky Code

A Mixed-Method Study


This web page presents the supplementary material of the paper Revealing Design Problems in Stinky Code - A Mixed-Method Study.

Authors: Willian Oizumi, Leonardo Sousa, Alessandro Garcia, Roberto Oliveira, Anderson Oliveira, O.I. Anne Benedicte Agbachi, Carlos Lucena

Following sections present the complementary information regarding the paper.

Abstract

Software design comprises the overall organization of source code into macro structures like components and interfaces. Unfortunately, the maintainability of a system may be hampered due to the unintended introduction of software design problems. Due to such negative eect, design problems should be properly identied and removed to maintain the evolvability of long-lived systems. To identify design problems, developers can rely on the analysis of code smells, which are surface symptoms of design problems. Since each code smell is only a partial symptom of a design problem, researchers proposed a strategy that agglomerates inter-related code smells to reveal the full extension of design problems. Nevertheless, there is little empirical evidence about the practical use of agglomerated code smells for the identication of design problems. Therefore, in this paper we assessed a strategy called code smell agglomeration. This evaluation was based on quantitative and qualitative analysis. For the quantitative analysis, we performed an experiment with 11 professionals, comparing the use of agglomerations to the traditional use of code smells. On the other hand, in the qualitative analysis, we conducted a post-experiment survey and an evaluation based on the semiotic engineering. Results indicate that agglomerations can help developers to nd more design problems with fewer false positives. In addition, our qualitative analysis revealed that agglomerations can yield even better results. To make it possible, diculties faced by developers must be tackled. For instance, we observed that (1) the information provided by an agglomeration would be better used if it were synthesized into an unique and concise representation, (2) textual information of agglomerations should use a simpler terminology to be more easily understandable, and (3) developers would benet from the implementation of an algorithm capable of suggesting possible design problems according to the symptoms revealed by an agglomeration.

Design Problems

We applied a quasi-experiment (Shadish, 2001). A quasi-experiment is an experiment in which the unit/groups are not assigned to conditions randomly. The experiment was conducted individually with each participant. They had to perform the experiment in two steps with four tasks in each one. Both steps comprise the same set of task, the only diference between the steps was regarding the usage of agglomerations. We divided the participants into two groups. The 1st group would identify design problems using agglomerations in the 1st step and after they would identify design problems using a list of non-agglomerated code smells in 2nd step. The second group of participants would make the identication inversely: using the non-agglomerated code smells in the 1st step and then using the agglomerations in the second step.

A design problem is a design characteristic that negatively impacts quality attributes such as maintainability and performance (Bar, 1998; Trifu, 2007). Design problems affect macro structures like components, interfaces, hierarchies and other structures that are relevant for the design of the system. We considered 8 types of design problems in our study. We opted by selecting these design problems since they are often considered as critical. Following we present the design problems used in the study.

Table I. Design Problem description
Name Description
Fat Interface Interface of a design component that offers only a general, ambiguous entry-point that provides non-cohesive services, thereby complicating the clients' logic.
Unwanted Dependency Dependency that violates an intended design rule.
Component Overload Design components that fulfill too many responsibilities.
Cyclic Dependency Two or more design components that directly or indirectly depend on each other.
Delegating Abstraction An abstraction that exists only for passing messages from one abstraction to another.
Scattered Concern Multiple components that are responsible for realizing a crosscutting concern.
Overused Interface Interface that is overloaded with many clients accessing it. That is, an interface with "too many clients".
Unused Abstraction Design abstraction that is either unreachable or never used in the system.

Categories of Agglomeration

Code smells are recurring micro structures in the source code that may indicate the manifestation of design problems (Fowler, 1999).

A code smell agglomeration is a group of inter-related code smells that may indicate the full extension of a design problem (Oizumi, 2016). The relations among code smells are determined by the relationships that exists between the smelly code elements. In our context, a smelly code element is a element that is affected by a code smell. Relations among smelly code may assume different forms, according to the agglomeration category. Below, on Table II, we present a brief definition of each category

Table II. Agglomeration Categories Definitions
Category Description
Intra-method Agglomerations of code smells that are located in single methods. There is an agglomeration in a method if the method is affected by at least Tmethod + 1 code smells. Tmethod is a threshold that can be defined by the developers.
Intra-class Just like methods, classes may also be affected by diverse code smells. There is an agglomeration in a class if the class is affected by at least Tclass + 1 code smells. Tclass is a threshold that can be defined by the developers.
Intra-component Agglomerations of code smells that occur inside of a single design component. This agglomeration comprises code elements that are located within a single component, and the elements are affected by the same type of code smell. The elements also must be connected by method calls or type references.
Hierachical Agglomerations composed by code elements that are affected by the same type of code smells, and these elements implement the same interface or inherit from the same code element.

Participant Selection

We needed participants that met the requirements described in Table III. We selected developers to identify design problems by sending a characterization questionnaire for a group of developers of our network. Their answers were analyzed to determine which of them were eligible to participate in the study based on the following requirements:

  1. R1. Four years or more of experience with software development and maintenance. We have chosen four years because this is the average time used by companies such as Yahoo (Yahoo, 2017) and Twitter (Twitter, 2017) to classify a developer as experienced.
  2. R2. No previous knowledge about Push Pull and Workflow Manager.
  3. R3. At least basic knowledge about code smells.
  4. R4. At least intermediary knowledge on Java programming and Eclipse IDE.

We included in the questionnaire a description of our definition of knowledge level, thereby allowing developers to have a similar interpretation of the possible answers. Table II presents the definition of knowledge level that developers received.

Table III. Knowledge classification
Classification Description
None I have never heard about it
Minimum I have heard about it, but I do not use it
Basic I have a general understanding, but I almost never use it
Intermediary I have a good understanding, and I use basic features sometimes
Advanced I have a deep understanding, and I often use advanced features
Expert I am a specialist in this topic, and I use many features almost every day
